# Spec 驱动开发

把想法变成可运行的代码，中间经过四份递进文档：

```
spec.md（做什么）→ plan.md（怎么做）→ tasks.md（按什么顺序做）→ checklist.md（做对了没）
```

每份文档在前一份基础上细化。每份都需要用户审批后才能进入下一阶段。

<HARD-GATE>
进入本 skill 的开发流程后，四份文档全部生成并获得用户批准之前，禁止创建或修改实现代码、测试代码、配置代码或生产文件。无论项目看起来多简单，一律走完流程；文档可以短，但不能跳过。
</HARD-GATE>

## 适用边界

本 skill 只在用户准备进入开发流程时生效。审批完成前允许：

- 阅读现有代码、文档和提交记录，理解上下文。
- 提问澄清需求，提出方案比较和风险分析。
- 起草或修改 spec.md、plan.md、tasks.md、checklist.md。
- 启动只读子代理进行文档评分和风险分析；子代理只能使用读取类工具和只读 git 命令，不得修改文件或运行会改变工作区的命令。
- 如果无法实际启动要求数量的子代理，停止当前阶段并告知用户；不得伪造子代理结果。
- 提供非落盘的示例说明，但不要把示例变成实现代码。

不要因为本流程阻止纯代码审查、解释现有代码、搜索定位、概念问答，或用户明确要求“只给建议、不改文件”的任务。

## 审批状态规则

- “段落确认”只表示当前小节可以继续，不等于整份文档批准。
- 每份文档必须在完整呈现后请求一次“整文档批准”。
- 只有用户明确表达“批准 spec.md / plan.md / tasks.md / checklist.md”“确认进入下一阶段”“这个文档通过”等，才视为该文档已批准。
- 用户只说“继续”“看下一段”“这段可以”，只能进入下一段，不能跨阶段。
- 如果用户提出修改，修改受影响内容后重新请求整文档批准。
- 小变更可以压缩每份文档的篇幅，也可以一次性呈现完整文档；但四份文档和四次整文档审批都必须存在。

## 文档位置和命名

开始前检查项目中是否已有 spec/plan/tasks/checklist 或 docs/ 下的相关文档。

- 若已有并与本任务相关，先读取并询问是更新、替换还是另建。
- 未经用户确认，不覆盖已有文档。
- 若项目已有文档目录或命名约定，优先沿用。
- 若无约定，默认使用 spec.md、plan.md、tasks.md、checklist.md。
- 已有文档即使被沿用，也必须在本任务上下文中重新确认有效，并获得对应的整文档批准；未经本轮明确批准，不得视为已批准文档。

## 反模式：「这个太简单了，不需要写 spec」

进入开发流程的每个项目都要走这套流程。一个工具函数、一次配置改动、一个单文件模块，全都要。越是「简单」的项目，未被审视的假设越多，返工的概率越高。文档可以写得短，但必须存在、必须被审批。

## 四份文档的职责

| 文档 | 回答什么 | 包含什么 |
|------|---------|---------|
| spec.md | 做什么 | 背景、目标、功能需求、非功能需求、边界（明确哪些留给后续）、验收标准 |
| plan.md | 怎么做 | 架构概览、组件划分、核心接口与数据结构、模块交互、技术决策 |
| tasks.md | 按什么顺序做 | 文件清单、有序任务列表、每个任务的步骤、验证方式和任务后子代理代码审查要求 |
| checklist.md | 做对了没 | 可观测的行为检查、集成检查、端到端场景 |

## 流程总览

```
用户想法
    │
    ▼
阶段一：需求澄清 ──→ spec.md ──→ 用户审批？
    │                                  │ 不通过：修改
    │                                  ▼ 通过
阶段二：技术设计 ──→ plan.md ──→ 用户审批？
    │                                  │ 不通过：修改
    │                                  ▼ 通过
阶段三：任务拆解 ──→ tasks.md ──→ 用户审批？
    │                                  │ 不通过：修改
    │                                  ▼ 通过
阶段四：验收设计 ──→ checklist.md ──→ 用户审批？
    │                                     │ 不通过：修改
    │                                     ▼ 通过
阶段五：开发（按 tasks.md 执行）
    │
    ▼
阶段六：验收（按 checklist.md 检查）
```

## 返工规则

- 如果只是措辞或验收项调整，修改当前文档并重新审批。
- 如果影响功能范围，回到 spec.md，更新并重新审批后再更新后续文档。
- 如果影响架构、接口或数据结构，回到 plan.md，更新并重新审批后再更新 tasks.md 和 checklist.md。
- 如果影响任务顺序、文件清单或验证方式，回到 tasks.md。
- 开发中发现需求不明确时，停止编码，回到对应文档阶段。
- 如果用户在审批完成前要求“直接写代码”“先实现再说”，说明当前阶段仍未通过；可以把文档写短，但不能跳过审批。

## 阶段一：需求澄清 → spec.md

**输入：** 用户的初步想法或粗略描述
**输出：** spec.md

### 步骤 1：了解上下文

阅读现有代码、文档和最近的提交记录，搞清楚当前状态和本次要做的新东西。

### 步骤 2：澄清需求

**一次只问一个问题。** 能用选择题就不用开放题。

关注点：
- 目的和动机——为什么要做这个？
- 成功标准——怎么判断做完了？
- 边界——哪些明确不做？
- 约束——性能、兼容性、安全性等

如果需求涉及多个独立子系统，立刻指出来，先帮用户拆分成子项目再深入细节。

### 步骤 3：提出需求范围方案

提出 2-3 种产品行为或需求范围方案，说清各自的优劣和你的推荐。推荐方案放第一个，解释为什么推荐它。技术实现方案留到 plan.md 阶段由多子代理提出。

### 步骤 4：分段呈现 spec

**逐段呈现**，每段确认后再展示下一段：

1. 背景与目标
2. 功能需求（F1, F2, ...）
3. 非功能需求（N1, N2, ...）
4. 不做的事
5. 验收标准

### spec.md 模板

```markdown
# [标题] Spec

## 背景
（要解决什么问题，当前已有什么）

## 目标
- ...

## 功能需求
- F1: ...
- F2: ...

## 非功能需求
- N1: ...

## 不做的事
- ...

## 验收标准
- AC1: ...
- AC2: ...
```

### 写作规则

- **聚焦行为描述。** 「提供一个主入口，接收环境上下文和可选配置，输出完整的 System Prompt 文本」——而非「提供 `BuildSystemPrompt(env, opts)` 函数」
- **保持语言无关。** 同一份 spec 应该适用于 Go、Java、Python和TypeScript等
- **方法名、类名、数据结构定义留给 plan.md**
- **保持实现无关的抽象层级**，具体文件路径留给 tasks.md
- **每个小节写完整**，所有内容就绪后再提交审批
- **每条功能需求至少对应一条验收标准**

### 自检

写完 spec.md 后逐项检查：

1. **占位符扫描**——有没有 TBD、TODO、未完成的小节？有就补上。
2. **语言泄漏**——有没有方法名、类型定义或特定语言的术语？有就删掉。
3. **歧义检查**——有没有哪条需求可以被理解成两种意思？有就选一种，写明确。
4. **范围检查**——是否聚焦到一个实现周期能完成的范围？
5. **验收覆盖**——每条 F 需求是否都有对应的验收标准？

发现问题就地修正，然后进入评分门禁。

### 评分门禁

自检通过后，启动 **3 个只读子代理**对 spec.md 独立评分（满分 100）。评分维度：
- **需求完整度**：背景、目标、功能需求、非功能需求、边界和验收标准是否齐全？
- **验收覆盖度**：每条 F 需求是否都有对应的可观测验收标准？
- **语言无关性与清晰度**：有无语言泄漏或歧义？

三个分数必须**分别达到 80 分**才能进入用户审批。任一低分，修改文档后重新评分；如无法启动 3 个子代理，停止当前阶段并告知用户。

### 用户审批

> spec.md 已生成，3 个子代理评分均 ≥80。请 review：
> - 功能需求是否完整？
> - 有没有遗漏的边界情况？
> - 「不做的事」是否合理？
> - 验收标准是否可观测？
>
> 确认后进入技术设计阶段。

**等待用户明确批准。** 如果用户要求修改，修改后重新评分并请求整文档批准。

## 阶段二：技术设计 → plan.md

**输入：** 已批准的 spec.md
**输出：** plan.md

### 流程

1. 重新阅读已批准的 spec.md，并重新了解当前项目结构、现有模式和约束。
2. 制定技术方案：优先满足 spec，贴合现有代码风格，风险最小，验证清晰，避免过度设计。
3. 定义核心数据结构和接口，画出模块间交互和数据流，记录关键技术决策及其理由。
4. **逐段呈现**，每段获得用户确认；完整 plan.md 呈现后先做自检，再启动评分门禁，通过后请求整文档批准。

### plan.md 模板

```markdown
# [标题] Plan

## 架构概览
（组件/模块划分，每个组件一段话）

## 核心数据结构

### [结构体名]
（字段定义及说明）

### [接口名]
（方法签名及用途）

## 模块设计

### [模块 A]
**职责：** ...
**对外接口：** ...
**依赖：** ...

### [模块 B]
...

## 模块交互
（调用链、数据流。哪个模块调哪个，什么顺序。）

## 文件组织
```
project/
├── src/
│   ├── feature-entry.[ext]   — 功能入口
│   ├── feature-core.[ext]    — 核心逻辑
│   └── feature-test.[ext]    — 测试
└── ...
```

> 文件组织示例必须根据当前项目语言、框架和目录结构改写，不得照抄。

## 风险与回滚

- 可能破坏的现有行为：...
- 安全/权限/隐私影响：...
- 数据兼容性或迁移影响：...
- 回滚方案：...

## 技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| ... | ... | ... |
```

### 写作规则

- **数据结构和方法签名在这一层定义**
- **说清架构如何满足 spec 的每条 F 需求**
- **文件组织写到目录和文件级别**
- **技术决策同时写明选择和理由**
- **风险与回滚必须具体**——涉及认证、权限、配置、网络、数据、缓存或外部服务时，明确安全影响和回滚方式
- **本文档与语言相关**——根据用户选择的语言来生成

### 自检

1. **spec 覆盖**——spec 的每条 F 需求是否都在架构中有归属？列出缺口。
2. **接口完整性**——光看接口描述，能不能独立实现每个模块？
3. **依赖清晰度**——模块间的依赖是否明确且无环？
4. **矛盾检查**——有没有技术决策和 spec 需求冲突？

### 评分门禁

自检通过后，启动 **3 个只读子代理**对 plan.md 独立评分（满分 100）。评分维度：
- **spec 覆盖度**：每条 F 需求是否都有架构归属？
- **接口与数据结构清晰度**：能否据此独立实现每个模块？
- **风险与回滚完整性**：是否覆盖认证、配置、外部依赖等关键风险？

三个分数必须**分别达到 80 分**才能进入用户审批。任一低分，修改文档后重新评分；如无法启动 3 个子代理，停止当前阶段并告知用户。

### 用户审批

> plan.md 已生成，3 个子代理评分均 ≥80。请 review：
> - 架构划分是否合理？
> - 核心接口定义是否完整？
> - 模块间交互是否清晰？
> - 技术决策是否认同？
>
> 确认后进入任务拆解阶段。

**等待用户明确批准。** 如果用户要求修改，修改后重新评分并请求整文档批准。

## 阶段三：任务拆解 → tasks.md

**输入：** 已批准的 spec.md + plan.md
**输出：** tasks.md

### 流程

1. 重新阅读 spec.md 和 plan.md
2. 列出文件清单——要创建、修改、测试哪些文件
3. 把 plan.md 的组件拆成有序任务
4. 每个任务是**一个聚焦、可独立验证的工作单元**，通常 5-15 分钟可完成；复杂任务拆到能明确验证、能安全回滚的粒度
5. 每个任务带有明确的验证方式
6. 每个任务必须写明：任务实现并验证通过后，至少启动三个只读子代理审查该任务相关变更（包括代码、测试、配置、脚本、生成文件和文档）；三个子代理应从不同角度审查，例如正确性/回归、代码质量/架构一致性、测试/错误处理/安全风险
7. 呈现给用户审批

### tasks.md 模板

````markdown
# [标题] Tasks

## 文件清单

| 操作 | 文件 | 职责 |
|------|------|------|
| 新建 | `src/feature-entry.[ext]` | 功能入口 |
| 新建 | `src/feature-core.[ext]` | 核心逻辑 |
| 新建/修改 | `src/feature-test.[ext]` | 测试或验证代码 |

## T1: [任务名]

**文件：** `path/to/file`
**依赖：** 无
**步骤：**
1. 定义该任务需要的最小接口或入口
2. 实现核心行为，并保持与 plan.md 中的设计一致
3. 补充该任务对应的测试或可运行验证

**验证：** 运行项目对应的构建或测试命令，期望该任务涉及的模块通过

**任务后审查：** 验证通过后，启动至少 3 个子代理从不同角度审查本任务相关代码变更；若无法启动 3 个子代理，停止执行并告知用户，不得标记本任务完成。

## T2: [任务名]

**文件：** `path/to/file`
**依赖：** T1
**步骤：**
1. ...

**验证：** 运行单元测试，环境字段正确填充

**任务后审查：** 验证通过后，启动至少 3 个子代理从不同角度审查本任务相关代码变更；若无法启动 3 个子代理，停止执行并告知用户，不得标记本任务完成。

## 执行顺序

```
T1 → T2 → T3
            ↘
T4（可并行）→ T5 → T6
```
````

### 写作规则

- **文件路径可以写**——这是实现层，需要具体
- **定位到文件级别**，用描述说明改动位置（行号会在下一次编辑后过期）
- **每个任务必须有「验证」部分**——「运行 X，期望看到 Y」
- **每个任务必须有「任务后审查」部分**——验证通过后至少启动 3 个子代理审查该任务相关代码变更；三个子代理应覆盖不同角度；无法启动足量子代理时不得标记任务完成
- **每个任务自包含**，写清楚完整细节（执行者可能不按顺序读）
- **每个步骤写具体操作**，所有内容就绪
- **依赖关系必须明确**——如果 T3 依赖 T1，写出来
- **粒度按可验证性拆分**——不要按时间机械拆分；按可验证的行为或文件级变更拆分

### 自检

1. **plan 覆盖**——plan.md 的每个组件是否至少有一个任务？
2. **占位符扫描**——有没有模糊的步骤或「类似 TX」的引用？
3. **依赖链**——是否存在合法的执行顺序，没有循环依赖？
4. **验证完整性**——每个任务是否都有具体的验证步骤？
5. **任务后审查完整性**——每个任务是否都写明验证通过后至少由 3 个子代理从不同角度审查代码，以及无法启动足量子代理时的停止规则？
6. **类型一致性**——函数名/类型名和 plan.md 定义的是否一致？

### 评分门禁

自检通过后，启动 **3 个只读子代理**对 tasks.md 独立评分（满分 100）。评分维度：
- **plan 覆盖度**：plan.md 的每个组件是否都有对应任务？
- **任务可验证性**：每个任务是否都有明确的验证方式和任务后审查要求？
- **依赖与顺序正确性**：依赖链是否合法、无循环？

三个分数必须**分别达到 80 分**才能进入用户审批。任一低分，修改文档后重新评分；如无法启动 3 个子代理，停止当前阶段并告知用户。

### 用户审批

> tasks.md 已生成，共 N 个任务，3 个子代理评分均 ≥80。请 review：
> - 任务粒度是否合适？
> - 依赖关系是否正确？
> - 有没有遗漏的实现步骤？
>
> 确认后进入验收设计阶段。

**等待用户明确批准。** 如果用户要求修改，修改后重新评分并请求整文档批准。

## 阶段四：验收设计 → checklist.md

**输入：** 已批准的 spec.md + plan.md + tasks.md
**输出：** checklist.md

### 流程

1. 重新阅读 spec.md 的验收标准——每条至少变成一个 checklist 条目。
2. 重新阅读 plan.md——提取集成、架构、风险与回滚层面的验证点。
3. 补充编译/测试/lint 检查。
4. 至少加一个端到端场景。
5. 先做自检，再启动评分门禁，通过后呈现给用户审批。

### checklist.md 模板

```markdown
# [标题] Checklist

> 每一项通过运行代码或观察行为来验证，聚焦系统行为。

## 覆盖来源

- spec.md 验收标准：已覆盖 / 未覆盖说明
- plan.md 风险与回滚：已覆盖 / 未覆盖说明

## 实现完整性
- [ ] [组件 A] 已实现且可被调用（验证：编译通过）
- [ ] [功能 X] 输出符合预期（验证：用示例输入运行，观察输出）

## 集成
- [ ] [模块 A] 正确调用 [模块 B]（验证：集成测试通过）
- [ ] 所有公开接口至少被一个真实调用方使用（验证：编译 + 全部测试通过）

## 编译与测试
- [ ] 项目编译无错误
- [ ] 所有单元测试通过
- [ ] lint 检查通过（如有配置）

## 回归与风险检查
- [ ] 关键旧行为未被破坏（验证：运行相关回归测试或手动场景）
- [ ] 安全/权限相关路径已验证（验证：用允许和拒绝场景分别运行）
- [ ] 如涉及配置、数据或外部服务变更，回滚路径已确认（验证：说明并演练可行的回滚步骤）

## 端到端场景
- [ ] 场景 1：[用户操作] → [可观测的预期结果]
- [ ] 场景 2：[边界情况] → [预期行为]
```

### 写作规则

- **可观测**——每一项都是「做 X，看到 Y」或「运行 X，期望 Y」
- **与实现解耦**——代码重构但行为不变时，checklist 依然适用
- **聚焦行为检查**——通过运行、观察、对比输出来验证
- **验证粒度对准功能和行为**（如「编译通过」「输出符合预期」）
- **至少一个端到端场景**——测试完整的用户可见流程
- **每一项附带验证方式**——写在括号里

### 自检

1. **spec 对齐**——spec.md 的每条验收标准是否都有对应的 checklist 条目？
2. **可观测性**——每一项是否都能不用逐行读代码就能验证？
3. **耦合测试**——如果重命名文件或移动函数，会不会有条目失败？有就重写那条。
4. **端到端**——是否至少有一个走完整流程的场景？
5. **风险覆盖**——涉及安全、权限、配置、数据或外部服务时，是否有回归、风险和回滚检查？

### 评分门禁

自检通过后，启动 **3 个只读子代理**对 checklist.md 独立评分（满分 100）。评分维度：
- **spec 覆盖度**：每条验收标准是否都有对应 checklist 条目？
- **可观测性**：每条是否都有明确的「运行 X，期望 Y」验证方式？
- **风险与端到端**：是否覆盖回归、风险路径和至少一个端到端场景？

三个分数必须**分别达到 80 分**才能进入用户审批。任一低分，修改文档后重新评分；如无法启动 3 个子代理，停止当前阶段并告知用户。

### 用户审批

> checklist.md 已生成，3 个子代理评分均 ≥80。请 review：
> - 是否完整覆盖了 spec 的验收标准？
> - 每项是否都可以运行/观察验证？
> - 端到端场景是否合理？
>
> 确认后进入开发阶段。

**等待用户明确批准。** 如果用户要求修改，修改后重新评分并请求整文档批准。

## 阶段五：开发

四份文档全部通过审批，开始实现。

**宣告：** 「四份文档已全部通过审批，按 tasks.md 开始开发。」

### 流程

1. 读 tasks.md，为所有任务创建进度追踪
2. 按执行顺序逐个完成任务：
    - 按步骤执行
    - 运行该任务的验证步骤
    - **先有证据再下结论**——先跑命令、看输出，再报状态
    - 验证通过后，启动至少 3 个子代理从不同角度审查该任务相关代码变更，例如正确性/回归、代码质量/架构一致性、测试/错误处理/安全风险
    - 处理子代理审查发现；确认无阻塞问题，或把不采纳/延后处理的发现明确记录给用户
    - 验证通过且任务后审查完成后才标记完成
3. 如果无法启动至少 3 个子代理审查当前任务：停止执行并告知用户；不得伪造审查结果，不得标记该任务完成
4. 如果被阻塞：停下来问，不要猜
5. 所有任务完成后进入阶段六

### 规则

- 按 tasks.md 的步骤执行，除非被阻塞否则不自由发挥
- 每个任务完成后必须跑验证，「应该没问题」不算证据
- 验证不通过就先修，修好再往下走
- 每个任务验证通过后必须启动至少 3 个子代理审查该任务相关代码变更；审查未完成前，不得把任务标记为完成
- 子代理审查发现必须处理、修复或明确记录为不采纳/延后处理；不得静默忽略高影响发现
- 每个任务或每组逻辑相关的任务完成后，整理验证证据、子代理审查结论和变更摘要；只有在用户明确授权或项目规范明确要求时才提交代码

## 阶段六：验收

### 流程

1. 读 checklist.md
2. 逐项执行：
    - 运行验证方式
    - 记录实际结果和证据（命令输出、观察到的行为）
    - 标记通过/不通过
3. 如果有不通过的：修复、重新验证、报告
4. 向用户呈现最终报告

### 规则

- **先有证据再下结论。** 先跑命令，看输出，然后再报告。
- 报告**实际结果**，不是预期结果。
- 有不通过的条目不丢人——说明 checklist 发挥了作用。修好重跑即可。

### 验收报告

```
## 验收报告

### 通过（N/M）
- [x] 条目 1 — 证据：...
- [x] 条目 2 — 证据：...

### 未通过（如有）
- [ ] 条目 3 — 预期：X，实际：Y，修复方案：...

### 端到端
- [x] 场景 1 — 结果：...
```

## 危险信号

出现以下想法时，停下来——你在为跳过流程找理由：

| 想法 | 现实 |
|------|------|
| 「这个太简单了，不需要写 spec」 | 越简单的项目，未被审视的假设越多 |
| 「我直接写代码就行」 | HARD GATE：四份文档全过了才能动代码 |
| 「spec 太明显了，直接跳到 plan」 | 「明显」意味着没被验证过，写出来让用户确认 |
| 「checklist 等做完了再补」 | checklist 决定了你要做什么，必须在编码前设计 |
| 「测试过了就说明没问题」 | 测试验证代码，checklist 验证需求 |
| 「应该没问题了」 | 跑一下。先有证据再下结论 |
| 「我知道用户想要什么」 | 问一下。一次一个问题 |

## 核心原则

- **一次一个问题**——每次聚焦一个问题
- **优先用选择题**——比开放题更容易回答
- **逐段确认，整文档审批**——逐步呈现，每段确认后再继续；完整文档呈现后必须单独请求整文档批准
- **层层递进**——每份文档在前一份基础上细化，逐步增加细节
- **YAGNI 铁律**——只设计和实现 spec 提到的内容
- **每个小节写完整**——所有内容就绪后再提交审批
- **先有证据再下结论**——先跑验证，再报告结果
- **聚焦行为描述**——spec 和 checklist 描述系统做什么
