# Spec 驱动开发

把需求、设计、任务、验收和实现授权分开管理，避免一次确认被扩大解释为后续写代码或发布许可。

## 不可覆盖的规则

1. 用户的最新明确指令优先。用户可以随时暂停、退出 `code-spec` 或切换到更快的流程；简要说明尚未完成的验证与风险后立即停止本 Skill，不得用 Skill 规则覆盖用户的新指令。
2. 退出本 Skill 只改变工作流，不会绕过仓库的 `AGENTS.md`、`CLAUDE.md`、安全规则、代码修改授权或其他适用 Skill。用户要求“快路径”也不自动构成这些额外授权。
3. 文档批准只批准文档。即使四份文档全部批准，也必须在写实现代码、测试、配置或生产文件前单独取得用户的明确代码修改授权。
4. 不执行 `git reset`、`git checkout`、`git clean`、`git revert`，不自动 stash，也不覆盖或丢弃用户已有改动。
5. 本 Skill 不负责 commit 或发布。普通本地 commit 请求按项目规则单独取得授权；任何 PR 的准备、push、创建或更新都必须移交 `submit-pr` Skill，不得自行复制发布流程。

## 启动与工作区基线

开始后先读取项目级指令，再做只读检查：

```powershell
git status --short
git diff -- <候选目标路径>
git diff --cached -- <候选目标路径>
```

记录完整 `git status` 基线。目标路径尚未确定时先记录全局基线，在每份计划得到具体文件清单后再次比对。

- 如果候选目标文件在基线中已有未提交、已暂存或未跟踪改动，视为可能与用户工作重叠：停止并报告路径、状态和拟议改动，等待用户澄清归属与继续方式。
- 其他路径的已有改动不属于本任务，不修改、不整理、不恢复。
- 每次准备写文件前重新检查目标路径；基线发生意外变化时停止，不猜测来源。

## 选择模式

先根据用户原话选模式；不确定时只问一个问题。

### 规划限定模式

当用户说“只写 spec/plan/tasks/checklist”“先拆任务”“暂时不要代码”或明确限定交付物时使用。

1. 只生成用户点名的文档；不要自动补齐其前置或后续文档。
2. 可以读取现有上游资料并声明假设，但缺失上游文档时不得擅自创建。
3. 对指定文档执行本 Skill 的版本绑定和风险分级评审；如果用户只要草稿且明确不要评审，记录其状态为“草稿、未批准”。
4. 指定文档完成后停止。不要进入下一文档、实现或发布。

### 完整实现模式

仅当用户明确要求完整 spec 驱动实现，或重大开发任务确实需要完整流程且用户确认进入时使用：

```text
spec.md -> plan.md -> tasks.md -> checklist.md -> 单独代码修改授权 -> 实现 -> 验收
```

在前一文档当前版本通过评审并获用户明确批准前，不进入下一文档。小篇幅不等于可以跳过阶段。

## 必须按需读取的参考

- 准备创建或修改任一规划文档前，读取 [references/document-templates.md](references/document-templates.md)，只使用当前文档对应的模板。
- 判断风险级别、启动 reviewer、计算评分、记录批准状态或处理 reviewer 异常前，读取 [references/review-rubric.md](references/review-rubric.md)。其中的评分表、状态表和提示词是本 Skill 的组成部分。
- 准备代码审查或记录任务证据前，从仓库根目录运行 `node .claude/skills/code-spec/scripts/change-set-hash.mjs -- <任务目标路径...>`；该只读工具对当前任务范围内的 HEAD、tracked diff、状态和 untracked 内容生成确定性 `change_set_hash`。

## 文档版本与依赖绑定

文档完成后用 Git 的对象算法计算内容 hash，不用时间戳或文件修改时间代替：

```powershell
git hash-object -- <文档路径>
```

每次评审和每次用户批准都记录：

- 文档路径与当前 `document_hash`
- 所有上游文档的路径与 `upstream_hashes`
- 风险级别、reviewer 数量、评审轮次和结论
- 用户批准原话；没有明确批准就保持未批准

状态只对精确 hash 组合有效：

1. 文档内容发生任何变化，立即使该文档旧评分和批准失效；重新计算 hash、自检、评审并请求批准。
2. 任一上游 hash 变化，无条件使全部下游文档的评分和批准失效，即使下游文本一字未改。按顺序重新绑定、评审并批准 `plan.md`、`tasks.md`、`checklist.md`。
3. 进入下一阶段前重新计算当前 hash，并与记录值逐项比较；不一致就停止阶段推进。

## 风险分级与 reviewer 数量

使用 `references/review-rubric.md` 的判定表：

- **低风险：1 个 reviewer。** 单模块、非敏感、容易回滚且不改变公共契约的文档或任务默认使用一个只读 reviewer。
- **高风险：3 个 reviewer。** 跨模块、安全或权限、认证、数据迁移、公开接口、生产配置、外部服务、难回滚或重大架构变更使用三个独立只读 reviewer。
- 用户明确要求严格模式时，无论原风险级别都使用三个 reviewer。

不要为了形式固定执行 12 次评审。风险变化时重新分级并记录原因。

## reviewer 只读与完整性校验

优先使用平台级只读 agent 或严格的读取工具白名单；不能只在提示词里声称“只读”。评审期间禁止并发写入。

如果平台无法强制限制工具，不得在共享工作树中运行 reviewer；目标路径 hash 加 `git status` 无法检测范围外文件或已 dirty 文件的内容变化，不能作为安全替代。可以在用户批准后把精确评审输入放进一次性隔离副本，在副本内运行 reviewer，并在前后用 change-set manifest 校验完整副本；副本产生任何变化都作废结果并丢弃副本。无法提供平台只读限制或隔离副本时停止评审并报告阻塞。

## 评分门禁与有限返工

每个被评审文档使用统一门禁。单个 reviewer 通过必须同时满足：

- 总分 `>= 80`
- 四个分项均有效，且分项之和严格等于总分
- `blockers` 为空

`must_fix` 必须在评审证据中列出，但不阻止推进，可在下一阶段跟进处理。

所有要求数量的 reviewer 都通过，当前 hash 才可请求用户批准。高分不能覆盖 blocker，也不得请求用户批准仍有阻塞项的版本。

收到评审意见后：

1. 修改文档即使旧评分失效；对新 hash 重新执行完整评审，不做局部补评。
2. 每个文档阶段最多自动返工 **2 轮**。不得无限启动子代理。
3. 两轮后仍未通过或 reviewer 意见互相冲突时停止自动修改，向用户列出分歧、影响、风险和可选路径，请用户裁决范围、方案或是否停止。
4. 用户裁决不会伪造通过结果；若继续当前门禁，仍需让新版本满足无 blocker 的通过条件。

## 四个文档阶段

### 1. spec.md

澄清目标、用户可观察行为、范围外事项、约束和可验证验收标准。一次只问一个必要问题。保持实现无关，每条功能需求至少对应一条验收标准。

### 2. plan.md

基于已批准且 hash 匹配的 spec 设计最小可行方案。写清组件边界、接口和数据流、目标文件、兼容性、错误处理、验证、风险与回滚。优先沿用仓库现有模式。

### 3. tasks.md

基于当前 spec/plan hash 拆成有序、聚焦且可独立验证的任务。每个任务写明目标文件、依赖、测试先行步骤、验证命令或可观察证据，以及按风险分级的任务后代码审查要求。

### 4. checklist.md

把 spec 验收标准、plan 风险和 tasks 交付项转成可运行或可观察的检查项。覆盖构建、测试、回归、权限/安全、回滚和至少一个端到端场景；每项都写明验证方式。

每阶段都执行：读取对应模板 -> 检查上游 hash -> 完整起草 -> 自检 -> 计算 hash -> 风险分级评审 -> 有限返工 -> 记录当前 hash -> 请求整文档明确批准。

## 独立取得代码修改授权

四份文档均以当前 hash 获批后，先重新检查工作区基线和目标路径，再向用户展示：

- 将修改、新建或删除的文件
- tasks 与 checklist 的验证范围
- 当前四份文档 hash
- 已知风险和回滚方式

然后单独询问是否授权现在修改实现代码、测试、配置及其他列出的目标文件。此前的“文档通过”“继续”“方案可以”都不算代码修改授权。未得到明确授权时停止在文档交付状态。

## 实现：测试先行和稳定版本循环

得到授权后按 tasks 顺序执行。每个任务：

1. 重新确认目标文件未与基线中的用户 dirty 改动重叠。
2. 先写会失败的测试或可执行验证，运行并确认它因目标行为尚未实现而失败。纯文档或无法执行的配置任务使用先失败的静态校验或明确的行为检查，并记录理由。
3. 写满足该任务的最小实现，运行任务验证和相关回归测试，保存实际输出。
4. 计算当前任务文件 hash，并运行 `node .claude/skills/code-spec/scripts/change-set-hash.mjs -- <任务目标路径...>` 取得确定性的变更集 hash；对当前版本启动风险分级的只读代码审查。
5. 只有验证通过、审查通过且证据绑定当前 hash 时才可标记任务完成。

任何审查后的代码、测试、配置、脚本或文档修改，都会让该任务原验证与审查证据同时失效。重新计算 hash，从任务验证开始重跑，并对修改后的完整 diff 重新审查；循环直到 hash 稳定。每个任务最多自动返工 **2 轮**；两轮后仍未通过时停止，展示失败证据、reviewer 分歧与修改选项，等待用户决定改变范围、返回文档阶段或终止任务。只允许对当前 hash 标记完成。

当前 hash 存在 `blocker` 时不得标记完成，用户接受风险不能把失败结论改写成通过。`must_fix` 必须记录在证据中，但不阻止完成。用户可以改变范围、返回相应文档阶段或终止任务；范围或文档变化后必须生成新 hash、重新验证并通过审查。仓库级安全禁令始终不可绕过。

## 最终验收与返修

逐项执行 checklist，记录命令、实际结果和当前变更集 hash。

- 某项失败需要改代码时，不得在验收阶段“顺手修完”后直接出报告。找到对应 task，将其重新打开，按测试先行流程重新验证并完成当前 diff 的代码审查。
- 任务重新稳定后，更新 hash 和证据，重跑受影响 checklist、依赖路径和必要回归；再继续验收。
- 最终报告列出通过项、未通过项、证据、当前 hash、已知风险和用户风险决定。

验收完成后停止。本 Skill 不提交或创建 PR；需要发布时明确移交 `submit-pr` Skill。

## 完成前自检

- 当前模式与用户最新指令一致；规划限定模式只交付点名文档。
- 工作区基线已记录，目标路径没有未经确认的 dirty 重叠。
- 每份有效评分和批准都绑定当前文档及上游 hash。
- reviewer 数量符合风险级别，且只读限制或前后完整性校验已有证据。
- 所有评分满足总分、分项合计和 blocker 三项条件；must_fix 已在评审证据中列出。
- 自动返工未超过两轮；未解决分歧已交给用户裁决。
- 代码修改有独立明确授权；每项任务都有测试先行、当前 hash 验证和审查证据。
- 验收返修已返回对应任务；发布请求已移交 `submit-pr`。
