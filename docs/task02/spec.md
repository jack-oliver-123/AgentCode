# AgentCode 纯对话 TUI 首版 Spec

## 背景

AgentCode 是一个从零开始构建的命令行 AI 编程助手项目，目标是逐步实现类似 Claude Code 的能力。本任务是第二个任务，也是第一个可运行产品切片：让用户能在终端启动 AgentCode，并进入一个可持续对话的 TUI 界面。

当前阶段只实现纯对话能力：用户输入问题，AgentCode 调用大模型，以流式方式逐字/逐片段展示回复，并在同一会话内保留上下文，使 AI 能基于前文继续回答。

## 目标

- 用户在终端输入 `agentcode` 后进入交互式对话界面。
- 用户可以连续输入多轮问题，AI 能记住当前会话中的前文。
- AI 回复必须以流式方式显示，而不是等待完整结果后一次性输出。
- 支持 Anthropic 协议和 OpenAI 协议，并允许通过配置切换。
- Provider 层对外呈现统一行为，为后续加入新模型后端保留扩展空间。
- Claude extended thinking 可以通过配置启用，但首版 TUI 默认不展示 thinking 内容，只展示最终回答。

## 功能需求

- F1: 启动入口  
  用户在终端执行 `agentcode` 后，系统进入 AgentCode 的交互式对话界面。

- F2: 核心 TUI 交互  
  界面应提供清晰的输入区和对话展示区，支持用户输入问题、提交问题、查看 AI 回复，并在首版优先保证核心交互可用，不追求完整复刻 Claude Code 的所有视觉细节、快捷键和主题能力。

- F3: 流式回复  
  AgentCode 调用大模型后，应随着服务端返回内容逐字或逐片段展示 AI 回复，不能等完整回复生成完后再一次性显示。

- F4: 多轮对话上下文  
  同一次启动会话中，AgentCode 应保存用户消息和 AI 回复，并在后续请求中带上必要上下文，使 AI 可以基于之前的对话继续回答。

- F5: Provider 协议切换  
  AgentCode 应支持 Anthropic 和 OpenAI 两种 API 协议，并通过配置文件选择当前使用的协议、模型、请求地址和认证信息。

- F6: 第三方中转站兼容  
  AgentCode 不应把请求地址固定为官方 API 地址；用户可以通过配置中的请求地址使用兼容 Anthropic 或 OpenAI 协议的第三方中转站。

- F7: 统一 Provider 行为  
  不同协议的模型后端应对上层对话流程暴露统一的流式响应行为，避免 TUI 直接依赖某一家供应商的请求和响应细节。

- F8: YAML 配置文件  
  AgentCode 使用 `.agentcode/config.yaml` 管理 LLM 供应商信息。配置文件至少包含四个核心字段：`protocol`、`model`、`base_url`、`api_key`。

- F9: 配置文件查找顺序  
  启动时应优先读取当前项目目录下的 `.agentcode/config.yaml`；如果不存在，再读取用户目录下的全局配置文件 `~/.agentcode/config.yaml`。

- F10: `.agentcode` 运行目录约定  
  AgentCode 运行相关的项目级配置和未来扩展文件应统一放在当前项目的 `.agentcode/` 目录下，避免在项目根目录散落多个 AgentCode 专用文件。

- F11: Claude extended thinking  
  当使用支持 extended thinking 的 Claude 模型时，AgentCode 应允许通过配置启用 extended thinking 请求能力；首版 TUI 默认隐藏 thinking 内容，只展示最终回答。

- F12: 明确排除 Agent 工具能力  
  本任务不实现 tool use、文件读取、文件编辑、命令执行、代码修改、MCP 或插件能力；AgentCode 在本阶段只做纯对话。

## 非功能需求

- N1: 配置安全  
  API key 只能来自用户配置，不应硬编码在代码或文档示例中；错误输出不得泄露完整 API key。

- N2: 可扩展性  
  Provider 抽象应支持后续添加新的协议或模型后端，而不要求重写 TUI 和会话流程。

- N3: 可观测错误  
  当配置缺失、配置格式错误、协议不支持、网络失败或模型服务返回错误时，界面应给出用户可理解的错误信息。

- N4: 跨平台命令行可用性  
  首版应面向常见终端环境运行，避免依赖只在单一操作系统可用的交互行为。

- N5: 响应体验  
  流式输出期间界面应能持续展示生成进度，避免长时间无反馈。

- N6: 范围克制  
  首版聚焦纯对话闭环，不引入工具调用、代码编辑、权限系统、长期记忆或复杂主题系统。

## 不做的事

- 不实现 tool use。
- 不读取、编辑或写入用户项目文件。
- 不执行 shell 命令或测试命令。
- 不实现代码修改、补丁生成或自动提交。
- 不实现 MCP、plugins、hooks、skills、subagents。
- 不实现长期记忆；只保留当前进程内的会话上下文。
- 不实现复杂快捷键系统、主题系统、布局自定义或完整 Claude Code 视觉复刻。
- 不实现模型列表拉取、模型自动选择或供应商账号管理。
- 不把 API key 写入仓库内示例配置。

## 验收标准

- AC1: 在终端执行 `agentcode` 后，可以看到 AgentCode 交互界面并输入问题。（覆盖 F1, F2）
- AC2: 输入一条真实问题后，界面能以流式方式逐字或逐片段显示 AI 回复，而不是等待完整回复后一次性输出。（覆盖 F3）
- AC3: 连续进行至少两轮对话时，第二轮问题可以引用第一轮内容，AI 回复能体现当前会话上下文。（覆盖 F4）
- AC4: 使用 Anthropic 协议配置时，AgentCode 能向配置中的请求地址发起流式请求并展示回复。（覆盖 F5, F6, F7, F8）
- AC5: 使用 OpenAI 协议配置时，AgentCode 能向配置中的请求地址发起流式请求并展示回复。（覆盖 F5, F6, F7, F8）
- AC6: 当当前项目目录存在 `.agentcode/config.yaml` 时，优先使用该配置；当项目配置不存在且 `~/.agentcode/config.yaml` 存在时，使用全局配置。（覆盖 F9, F10）
- AC7: AgentCode 的项目级运行相关文件约定在 `.agentcode/` 目录下，不要求用户在项目根目录放置零散的 AgentCode 专用配置文件。（覆盖 F10）
- AC8: 当配置缺失、字段缺失或协议不支持时，AgentCode 显示可理解的错误信息，并且不泄露完整 API key。（覆盖 N1, N3）
- AC9: 当 Claude extended thinking 被配置启用时，请求能力被传递给支持该能力的 Provider；首版 TUI 默认不显示 thinking 内容，只显示最终回答。（覆盖 F11）
- AC10: 首版运行过程中不会尝试读取/编辑项目文件、执行 shell 命令或调用工具能力。（覆盖 F12 和“不做的事”）
- AC11: Provider 抽象允许在不改写 TUI 对话流程的前提下增加新的协议后端。（覆盖 F7, N2）
