# Tool System Spec

## 背景

AgentCode 当前已经具备可运行的 TypeScript CLI、配置加载、OpenAI/Anthropic 流式 Provider、会话控制器和 Ink TUI，但模型回复仍局限于纯文本生成。用户提出项目相关问题时，模型不能读取仓库文件、搜索代码、修改文件或执行命令，因此 AgentCode 仍更接近聊天机器人，而不是真正能完成开发任务的终端 Agent。

task04 的目标是在不引入完整自动 Agent Loop 的前提下，为 AgentCode 增加第一版工具系统。模型可以在一次用户请求中选择一个工具，AgentCode 负责执行该工具，并把结构化结果回灌给模型，让模型基于真实执行结果给出最终回答。

## 目标

- 让模型能够通过统一工具机制请求一次本地能力调用。
- 提供六个核心工具：读取文件、写入文件、编辑文件、执行命令、按模式查找文件、搜索代码内容。
- 用集中注册机制管理工具元信息、参数约束、查找和 Provider API 暴露格式。
- 工具执行失败时返回结构化错误结果，而不是让会话崩溃。
- 支持流式解析 Provider 返回的工具调用参数碎片，并在参数完整后执行工具。
- 将工具执行结果写回对话上下文，再请求模型生成最终文本回答。
- 明确把多轮自动工具循环、复杂权限系统和多工具链式执行留到后续任务。

## 功能需求

- F1: AgentCode 应提供统一的工具抽象，每个工具都具备稳定名称、面向模型的描述、参数约束和可执行行为。

- F2: AgentCode 应提供工具注册中心，集中登记可用工具，并支持按名称查找工具。

- F3: AgentCode 应能把已注册工具转换成 Provider API 可识别的工具声明，使模型能够知道有哪些工具可用、每个工具需要什么参数。

- F4: AgentCode 应提供读取文件工具，允许模型读取工作区内指定文本文件内容，并在失败时返回清晰的结构化错误。

- F5: AgentCode 应提供写入文件工具，允许模型向工作区内指定路径写入内容，并在路径无效、写入失败或权限受限时返回清晰的结构化错误。

- F6: AgentCode 应提供编辑文件工具，使用“原文唯一匹配替换”更新文件内容；当原文匹配不到或匹配多次时，不应修改文件，并应返回可供模型重试的明确错误。

- F7: AgentCode 应提供执行命令工具，允许模型在当前工作区执行命令，并返回退出码、标准输出、标准错误和超时信息。

- F8: AgentCode 应提供按模式查找文件工具，允许模型用 glob 风格模式发现工作区内文件，并返回匹配路径列表。

- F9: AgentCode 应提供搜索代码内容工具，允许模型在工作区内按文本或正则搜索文件内容，并返回匹配文件、行号和匹配片段。

- F10: 工具执行应具备统一超时控制；超时不应导致整个会话崩溃，而应作为结构化工具结果返回给模型。

- F11: Provider 流式响应中出现工具调用时，AgentCode 应能解析工具名称和 JSON 参数碎片，在参数完整后执行对应工具。

- F12: 一次用户请求最多执行一个工具调用；工具结果回灌给模型后，AgentCode 应请求模型生成最终文本回答，并在该回答结束后停止本轮会话。

- F13: 当模型请求未知工具、参数不符合约束或工具执行失败时，AgentCode 应把失败作为工具结果回灌给模型，让模型有机会向用户解释或调整建议。

- F14: 工具执行结果不得泄漏已知敏感信息；错误、stdout/stderr、TUI 输出和测试快照中仍应遵守现有 secret redaction 约束。

## 非功能需求

- N1: 工具系统应保持当前纯终端 CLI/TUI 体验可用；没有工具调用时，现有纯文本对话行为不应回退。

- N2: 工具接口和注册中心应易于扩展，后续新增工具不应要求改动 Provider、TUI 或会话控制器的大量核心逻辑。

- N3: 工具参数校验应在执行前完成；参数无效时不应进入真实文件、命令或搜索操作。

- N4: 文件类工具应默认限制在当前工作区内，避免模型通过相对路径或绝对路径访问工作区外文件。

- N5: 写文件和改文件操作应避免静默覆盖不可预期内容；失败路径必须返回明确原因。

- N6: 命令执行工具应有默认超时时间，并能区分“命令执行失败”和“命令超时”。

- N7: 工具结果应采用稳定结构，方便模型理解，也方便测试断言；不依赖人类自然语言解析成功与否。

- N8: Provider 适配应兼容当前 OpenAI-compatible 和 Anthropic 两类协议的现有流式文本能力；即使某个 Provider 的工具调用能力暂不完整，也应以清晰错误或受限能力呈现。

- N9: 流式工具调用参数拼接应能处理 JSON 被拆成多个片段的情况；参数不完整或最终不是合法 JSON 时，应返回协议级错误或结构化工具错误。

- N10: 工具执行和 Provider 错误仍应复用现有公开错误模型，避免把内部异常、堆栈或敏感配置直接暴露到 TUI。

- N11: 这一阶段的实现应具备自动化测试覆盖，包括工具单元测试、Provider 工具调用解析测试、会话控制器集成测试和至少一个端到端 smoke 场景。

## 不做的事

- 不实现多轮自动 Agent Loop。模型拿到一次工具结果后，本轮只生成最终文本回答，不继续自动请求第二个工具。

- 不实现同一轮多工具并发或批量工具调用。一次用户请求最多执行一个工具调用。

- 不实现完整权限弹窗、允许/拒绝策略、权限记忆或 sandbox 策略。本阶段先定义清晰的工具执行边界和错误结果，权限系统留到后续任务。

- 不实现长期 memory、会话持久化、resume、计划模式、子代理、MCP、plugins、hooks 或 skills。

- 不实现二进制文件读取、富媒体文件解析或大文件智能分块。读取文件工具只面向文本内容，并可对输出大小做限制。

- 不实现复杂 shell 交互、后台任务、交互式命令输入或长期运行进程管理。执行命令工具只支持一次性命令，带超时并返回结果。

- 不实现 git 自动提交、自动推送、自动创建 PR 或外部服务发布操作。

- 不要求工具系统完全复刻 Claude Code 的内部协议或行为；只实现 AgentCode 当前阶段可验证、可扩展的最小闭环。

## 验收标准

- AC1: 在没有工具调用的普通对话中，AgentCode 仍能正常流式显示模型文本回复。

- AC2: 已注册工具能够被转换为 Provider API 可识别的工具声明，且声明中包含工具名称、描述和参数约束。

- AC3: 模型请求读取文件时，AgentCode 能返回工作区内目标文本文件内容；目标不存在、路径越界或读取失败时返回结构化错误结果。

- AC4: 模型请求写入文件时，AgentCode 能向工作区内目标路径写入内容；路径越界、写入失败或权限受限时返回结构化错误结果。

- AC5: 模型请求编辑文件时，只有原文在目标文件中唯一出现才会替换；匹配不到或匹配多次时文件保持不变，并返回明确错误。

- AC6: 模型请求执行命令时，AgentCode 能返回退出码、标准输出、标准错误和是否超时；命令超时时不会卡死会话。

- AC7: 模型请求查找文件时，AgentCode 能按 glob 风格模式返回工作区内匹配路径列表。

- AC8: 模型请求搜索代码内容时，AgentCode 能返回匹配文件、行号和匹配片段。

- AC9: Provider 流式返回工具调用参数碎片时，AgentCode 能拼接成完整 JSON 参数并执行对应工具。

- AC10: 当工具调用成功后，AgentCode 会把结构化工具结果回灌给模型，并显示模型基于工具结果生成的最终文本回答。

- AC11: 一次用户请求中，即使模型在工具结果回灌后继续请求工具，AgentCode 也不会执行第二个工具调用，并会以清晰错误结束或转为最终文本说明。

- AC12: 未知工具、无效参数、工具执行失败和工具超时都不会导致进程崩溃；用户能在 TUI 中看到安全的错误说明或最终回答。

- AC13: API key、Authorization header、Bearer token、JWT 和现有 sentinel secret 不会出现在工具结果、错误信息、stdout/stderr、TUI pane capture 或测试快照中。

- AC14: 自动化验证覆盖工具单元测试、Provider 工具调用解析测试、会话控制器工具回灌集成测试和 tmux/psmux E2E smoke 测试。
