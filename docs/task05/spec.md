# Agent Loop Spec

## 背景

AgentCode 在 task04 已实现 6 个内置工具和单工具调用闭环。当前 `ChatSessionController` 硬编码为"一次 provider 调用 → 最多一次工具执行 → 强制 toolChoice:none 出最终回答"的两步流程。用户每次只能得到一步工具结果，要手动催促模型继续，无法完成需要多步推理和多次工具调用的实际编程任务。

## 目标

将 AgentCode 从被动应答升级为自主 Agent——实现 ReAct 模式的 Agent Loop，让模型自主循环（推理 → 调工具 → 观察结果 → 继续推理），直到任务完成或触发停止条件。同时引入 Plan Mode，支持"先规划再执行"的两段式工作流。

## 功能需求

- F1: ReAct Agent Loop — 用户提交一条消息后，Agent 进入循环：调用 LLM → 如果 LLM 返回工具调用则执行工具 → 将工具结果回写到上下文 → 再次调用 LLM，直到 LLM 返回纯文本回答（不再请求工具）或触发停止条件。整个循环对用户表现为一次完整的"turn"。

- F2: 停止条件 — 循环在以下任一条件满足时终止：
  1. 模型返回纯文本、不再请求工具调用（正常完成）
  2. 达到迭代上限（默认 50 次，用户可通过配置调整）
  3. 用户主动取消（通过 abort/signal 机制）
  4. 连续 N 次调用不存在的工具（防止模型幻觉死循环）
  5. Provider 流式响应出错（不可恢复的错误）

- F3: 异步事件流 — Agent Loop 向外暴露一套异步事件流，事件类型包括：文本 delta、工具调用开始、工具执行结果、迭代进度（当前第几轮）、token 用量累计。TUI 通过订阅事件流驱动渲染，Agent Loop 本身不直接操作 UI。

- F4: 流式双路收集 — 每轮 LLM 响应采用双路处理：一路实时将 text delta 推送为事件（驱动 TUI 流式显示），另一路累积完整响应文本和工具调用信息，供循环判断下一步动作。

- F5: 多工具调用并发调度 — 当 LLM 一次返回多个工具调用时，按工具的 risk 级别分批执行：risk 为 `read` 的工具可并发执行，risk 为 `write` 或 `execute` 的工具串行执行。同一批内并发，不同批按顺序串行。

- F6: Plan Mode — 提供两段式工作流：
  - `/plan` 命令：Agent Loop 只注入 risk 为 `read` 的工具，模型自主循环产出结构化计划（步骤列表），计划存储在会话状态中，循环结束后向用户展示。
  - `/do` 命令：注入全部工具，将已存储的计划作为上下文注入，模型参考计划自主执行任务。计划是强参考而非强绑定，模型可根据执行中的发现灵活调整。
  - 结构化计划通过一个内置的 `submit_plan` 工具输出，schema 约束为步骤数组（每步有标题和描述）。

## 非功能需求

- N1: Agent 与 UI 解耦 — Agent Loop 作为独立模块运行，不依赖任何 UI 组件。所有状态变化通过事件流传递，TUI 只是事件的消费者之一。
- N2: 可取消性 — 循环中的任何阶段（等待 LLM 响应、执行工具）都必须响应 abort signal，取消后在当前步骤结束时干净退出，不留悬挂的 Promise 或资源泄漏。
- N3: 流式体验 — 每轮 LLM 推理的文本 token 必须实时推送到 TUI，不能等整轮结束才显示。工具执行期间显示工具名和进度状态。
- N4: 上下文正确性 — 循环中每次调 LLM 时，上下文必须包含本次 turn 内所有已完成的工具调用和结果，保证模型能看到完整的推理链。

## 不做的事

- 不做权限系统 — 工具执行前不弹出用户确认弹窗，不做分级授权，留给后续章节。
- 不做上下文压缩 — 循环中上下文只增不减，不做消息裁剪、摘要或滑动窗口，留给后续章节。
- 不做用户交互式确认 — 循环过程中不暂停等待用户输入确认，工具自动执行。
- 不做多会话/恢复 — 不支持 Agent Loop 中途退出后恢复到之前的循环状态。
- 不做 token 预算硬限制 — 统计 token 用量并上报事件，但不主动截断循环（迭代次数上限已经兜底）。

## 验收标准

- AC1: 用户发送一条需要多步工具调用的消息（如"读取 A 文件内容，写入 B 文件"），Agent 自主完成全部步骤并返回最终文本回答，中间无需用户再次输入。
- AC2: 循环达到迭代上限（配置为 3 次用于测试）时，循环终止并向用户报告"已达迭代上限，任务未完成"。
- AC3: 用户在循环中按取消键，循环在当前步骤结束后干净退出，无悬挂 Promise。
- AC4: 模型连续调用不存在的工具超过阈值时，循环终止并报告原因。
- AC5: Provider 流式响应出错时，循环终止并报告错误信息。
- AC6: 每轮 LLM 推理的文本 token 实时流式显示在 TUI 中，工具执行期间显示工具名状态。
- AC7: Agent Loop 向外发出的事件流包含：文本 delta、工具调用开始、工具执行结果、迭代进度、token 用量，TUI 通过事件驱动渲染。
- AC8: LLM 一次返回多个工具调用时，read 类工具并发执行，write/execute 类工具串行执行，全部结果正确回写上下文。
- AC9: `/plan` 命令触发 Agent Loop 仅注入 read 类工具，模型通过 `submit_plan` 工具输出结构化步骤列表，存储在会话状态中并展示给用户。
- AC10: `/do` 命令注入全部工具，已存储的计划作为上下文注入，模型参考计划自主执行任务。
- AC11: 循环中每次调 LLM 时，上下文包含本次 turn 内所有已完成的工具调用和结果（可通过日志或测试断言验证）。
